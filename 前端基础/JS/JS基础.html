说说你对闭包的理解 ​

  1. 概念

通过 chrome 开发者调试工具得知: 闭包本质是内部函数中的一个容器(非 js 对象), 这个容器中包含了引用的局部变量

  1. 产生原因
  • 函数嵌套
  • 内部函数引用外部函数的局部变量
  • 调用外部函数时就会产生闭包
  1. 闭包的生命周期

  • 产生：当内部函数创建时

  • 死亡：当内部函数没有变量引用，成为垃圾对象，就会自动被 GC(垃圾回收机制)自动回收

  1. 闭包作用：
  • 延长局部变量生命周期（活的久点）
  • 让函数外部可以间接操作函数内部的局部变量数据
  1. 闭包在项目中的应用
  • React 中复用函数的手段，高阶函数形式
  • 实际开发中很少使用闭包，我一般研究源码时发现闭包的场景，vue2 响应式原理中数据劫持阶段，内部通过闭包形式保存 dep 对象，这个 dep 对象在 vue2 响应式原理非常重要
    • 源码：https://github.com/vuejs/vue/blob/main/src/core/observer/index.ts 128 行开始

说说 JS 的垃圾回收机制 ​

在 JS 中对象的释放（回收）是靠浏览器中的垃圾回收器来回收处理的

  1. 垃圾回调器
  • 浏览器中有个专门的线程， 它每隔很短的时间就会运行一次
  • 主要工作：判断一个对象是否是垃圾对象，如果是，清除其内存数据，并标记内存是空闲状态
  1. 垃圾回收策略

JavaScript 引擎采用自动垃圾回收机制来管理内存，主要目的是自动追踪和释放不再使用的内存空间，防止内存泄漏。常见的垃圾回收策略主要有以下几种：

  1. 标记-清除算法（Mark-and-Sweep） 标记阶段：垃圾回收器从根对象（如全局变量、执行上下文栈中的变量）开始遍历，访问到的对象会被标记为“已访问”。 清除阶段：遍历完成后，未被标记的对象被视为垃圾，回收器将这些内存空间释放。 缺点：标记清除过程可能导致程序暂停（即所谓的“Stop-the-world”），影响性能；另外，频繁的内存回收可能导致内存碎片。
  2. 引用计数（Reference Counting） 每个对象都有一个引用计数，被引用一次，计数加一；引用被释放，则计数减一。 当对象的引用计数变为零时，表示该对象不再被使用，可以被回收。 优点：简单高效，实时回收。 缺点：无法解决循环引用问题，可能导致内存泄漏。
  3. 弱引用与循环引用处理 为了解决引用计数法中的循环引用问题，现代JavaScript引擎引入了弱引用（Weak Reference）的概念，以及改进的垃圾回收算法来检测并处理循环引用。
  4. 分代收集（Generational Collection） 将内存分为新生代（Young Generation）和老生代（Old/Tenured Generation）。 新创建的对象通常分配在新生代，如果对象经过几次垃圾回收后依然存活，则晋升到老生代。 新生代采用效率更高的算法（如复制算法），老生代可能使用标记-清除或增量标记-清除结合其他优化策略。 优点：提高垃圾回收效率，减少程序暂停时间。
  5. 增量标记与并发标记 增量标记：将标记过程分成多个小步骤执行，每次执行一小部分，以减少对主线程的影响。 并发标记：垃圾回收器在主线程执行代码的同时进行标记工作，进一步减少暂停时间。
  6. 空闲时间收集（Idle-time Collection） 垃圾回收器在程序空闲时执行回收工作，尽量减少对用户体验的影响。 总结 JavaScript引擎通常结合多种策略来优化垃圾回收过程，平衡性能与内存管理的效率。不同浏览器和JavaScript环境（如Node.js）的具体实现细节可能有所不同，但基本原理围绕上述几种策略展开。

    • 他们都是新的存储数据的结构

说说 ES6 的 promise ​

  1. 概念

异步代码解决方案，用于解决回调地狱问题。

  1. promise 对象内部有 3 种状态
  • pending 初始化状态
  • resolved / fulfilled 成功状态
  • rejected 失败状态

状态只能变化一次，只能有以下两种变化：

  • pending --> resolved
  • pending --> rejected

状态发生变化后不能在改了。

  1. 如何改变 promise 的状态
  • 调用 resolve(), 改成成功状态
  • 调用 reject(), 改为失败状态
  • throw new Error(), 改为失败状态
  1. promise 实例对象上的方法
  • then 接受两个回调（一般只接受一个），第一个是成功回调，第二个是失败回调
  • catch 接受一个回调，是失败回调
  • finally 接受一个回调，不管成功/失败都会触发
  1. Promise 构造函数上的方法
  • Promise.resolve() 返回一个成功的 promise 对象
    • 也可能返回失败的 promise 对象，比如 Promise.resolve(Promise.reject())
  • Promise.reject() 返回一个失败的 promise 对象
  • Promise.all([promise1, promise2, ...]) 只有所有 promise 成功才成功，只要有一个 promise 失败就会失败
  • Promise.allSettled([promise1, promise2, ...]) 只要所有 promise 状态发生变化就成功，结果值包含所有 promise 的结果值（不管成功/失败）
  • Promise.race([promise1, promise2, ...]) 只要有一个 promise 成功/失败，就成功/失败。
  1. 应用
  • 在项目中一般是使用 axios 发送请求时会使用，返回值是一个 promise 对象，结合 async await 来处理
  • 如果同时要发送多个请求的话，可以使用 Promise.all() 方法来处理

谈谈模块化语法 ​

  1. Commonjs 模块化语法
  • 主要用于 NodeJS 端
  • 语法：
    • 引入：require
    • 暴露：exports / module.exports
  1. ES6 模块化语法

  • 主要用于浏览器端

  • 语法：

  • 引入：import

  • 暴露：export

  • 总结

    • 如果模块采用默认暴露：import xxx from 'xxx'
    • 如果模块采用分别/统一暴露：
      • 如果需要引入模块部分内容：import { xxx } from 'xxx'
      • 如果需要引入模块全部内容：import * as xxx from 'xxx'

  • 扩展

    • 无论什么暴露方式，暴露的一定是一个对象
      • 默认暴露：对象上添加一个 default 属性，值为暴露的内容
      • 分别暴露/统一暴露：直接在对象上暴露内容
    • import { xxx } from './xxx' 引入暴露内容的某个属性（部分）
    • import xxx from './xxx' 引入暴露内容的 default 属性（只要 default）
    • import * as xxx from './xxx' 引入暴露内容的所有内容（全部）

浅度克隆和深度克隆 ​

  1. 浅度克隆
  • object.assign()
  • 扩展运算符:
  • Array.prototype.slice()
  • Array.prototype.concat()
  1. 深度克隆
  • JSON.parse(JSON.stringify())
  • 自定义深度克隆
  • lodash 中 cloneDeep
  1. 区别
  • 浅度克隆: 对当前对象进行克隆，基本类型克隆生成新值，引用类型克隆的是地址值（所以当修改对象内部引用类型数据时，原对象也会发生变化）
  • 深度克隆: 会完全复制整个对象，包括这个对象所包含的内部对象（所以不管如何修改对象数据，原对象都不会发生变化）。
  1. 深度克隆应用
  • 权限管理功能中使用深度克隆克隆了异步路由表

HTTP 主要版本功能 ​

  • HTTP/0.9
    • 极简协议：HTTP的初始版本，极其简单，仅支持GET请求方法，且不包含请求头和响应头，仅能传输HTML文档。
    • 无状态：HTTP/0.9就已经是无状态协议，意味着服务器不保存关于客户端的任何信息。
  • HTTP/1.0
    • 增加功能：引入了请求头和响应头，支持更多的请求方法（如POST、HEAD），增加了状态码和内容协商机制。
    • 持久连接：默认情况下，每个请求/响应后关闭TCP连接，但引入了Connection: keep-alive头部字段支持长连接，减少连接建立的开销。
    • 缓存控制：引入了缓存机制，通过Cache-Control、Expires等头部字段控制。
    • 无主机名：HTTP/1.0的请求中不包含Host字段，这限制了服务器在同一IP地址上托管多个网站的能力。
  • HTTP/1.1
    • 默认持久连接：默认开启长连接（Keep-Alive），减少了TCP连接的建立和关闭次数。
    • 管道化：允许在同一个TCP连接中发送多个请求，但服务器必须按照请求的顺序返回响应。
    • 新增状态码和头字段：增加了更多的状态码和请求/响应头字段，如Host头成为强制性字段，支持虚拟主机。
    • 分块传输编码：支持动态生成内容的分块传输，提高了响应的灵活性。
    • 请求方法扩展：引入OPTIONS、PUT、DELETE等更多请求方法。
    • 内容编码：支持GZIP等压缩算法，减少传输数据量。
  • HTTP/2
    • 二进制分帧：最显著的变化是将数据和控制信息分割成更小的帧，实现了多路复用，可以在同一连接上交错发送多个请求和响应，解决了HTTP/1.x的队头阻塞问题。
    • 头部压缩：使用HPACK压缩算法减少头部信息的大小，降低了传输成本。
    • 服务器推送：服务器可以主动向客户端推送资源，无需客户端明确请求。
    • 优先级和依赖：允许客户端指定请求的优先级，服务器可以根据优先级调度响应。
    • 加密传输：虽然HTTP/2本身不强制要求加密，但大多数实现都建议或要求使用TLS，形成了HTTPS。
  • HTTP/3
    • 基于QUIC：HTTP/3建立在QUIC（Quick UDP Internet Connections）协议之上，而非传统的TCP，QUIC使用UDP来改善连接延迟和安全性。
    • 更快的连接建立：QUIC的连接建立时间更短，因为它复用了之前的连接状态。
    • 更好的拥塞控制：QUIC提供了自己的拥塞控制机制，优化了在网络拥塞情况下的性能。
    • 加密：HTTP/3强制使用TLS加密，进一步提升了安全性。 每个新版本的HTTP都是在前一版本基础上针对网络环境的变化和需求的提升进行优化，以提供更高效、更安全的网络通信体验。

HTTP 和 HTTPS 协议的区别 ​

HTTP（Hypertext Transfer Protocol）和HTTPS（Hypertext Transfer Protocol Secure）都是用于在Web上发送和接收数据的协议，但它们之间存在关键的安全和隐私差异：

  • 数据传输安全性： HTTP：使用明文传输数据，不进行加密，这意味着数据可以被中间人（如黑客或ISP）轻易拦截和读取。 HTTPS：基于SSL/TLS（Secure Sockets Layer / Transport Layer Security）协议，它在客户端和服务器之间建立加密连接，保护数据免受窃听和篡改。
  • 数据完整性： HTTP：没有内置机制验证数据是否在传输过程中被修改。 HTTPS：通过数字证书和哈希算法确保数据在传输过程中保持完整，防止数据被篡改。
  • 身份验证： HTTP：无法确认服务器的身份，容易受到中间人攻击，用户可能被引导到假冒的网站。 HTTPS：通过SSL/TLS证书，服务器的身份得到验证，浏览器可以显示锁形图标，表明连接是安全的。
  • 浏览器显示： HTTP：URL通常以 http:// 开头，浏览器地址栏不显示安全标识。 HTTPS：URL以 https:// 开头，地址栏显示锁形图标，有时还显示绿色地址栏和证书信息，表示网站的身份已验证。
  • 信任度和用户信心： HTTP：用户可能对网站的信任度较低，特别是在涉及敏感信息（如登录凭证、信用卡信息）的场景。 HTTPS：提供更高的信任度，用户更愿意在这样的网站上输入个人信息。
  • 搜索引擎偏好： HTTP：搜索引擎可能降低未使用HTTPS的网站排名，因为它们被认为是不太安全的。 HTTPS：搜索引擎如Google倾向于优先索引和显示使用HTTPS的网站，认为它们更安全，有利于SEO。
  • 总的来说，HTTPS提供了比HTTP更高级别的安全性和隐私保护，是现代Web应用的标准配置，尤其是涉及到用户数据交换和敏感信息时。

TCP 的三次握手和四次挥手 ​

三次握手的目的：是为了确认双方的接收能力和发送能力是否正常。

第一次握手：由浏览器发送给服务器，服务器确定浏览器的发送能力 第二次握手：由服务器发送给浏览器，浏览器确定的服务器接受和发送能力 第三次握手：由浏览器发送给服务器，服务器确定的浏览器的接受能力

四次挥手的原因是因为 TCP 的连接是全双工的，所以需要双方分别释放到对方的连接，单独一方的连接释放，只代表不能再向对方发送数据，连接处于的是半释放的状态。（确保双方都接受完毕和发送完毕数据）

第一次挥手：由浏览器发送给服务器，告诉服务器数据发送完了 第二次挥手：由服务器发送给浏览器，告诉浏览器数据接受完了 第三次挥手：由服务器发送给浏览器，告诉浏览器数据发送完了 第四次挥手：由浏览器发送给服务器，告诉服务器数据接受完了，此时浏览器会自动断开链接，服务器也会

TCP（Transmission Control Protocol）协议的连接建立和断开过程分别通过三次握手（Three-Way Handshake）和四次挥手（Four-Way Handshake）来完成，这两个过程保证了数据传输的可靠性。

  • 三次握手（Three-Way Handshake）
    • 第一次握手：客户端发送一个带有SYN（同步序列编号，Synchronize Sequence Number）标志的TCP报文到服务器，并进入SYN_SEND状态，等待服务器确认。这个报文不包含应用层数据，仅用来同步序列号。假设客户端选择的初始序列号为x，则报文中的序列号为seq=x。
    • 第二次握手：服务器接收到客户端的SYN报文后，会以自己的SYN报文作为应答，并附上确认（ACK）标志。同时，服务器也会为自己选择一个初始序列号，假设为y，则服务器回应的报文包含SYN=1，ACK=1，seq=y，ack=x+1（确认客户端的序列号加1）。服务器进入SYN_RECV状态。
    • 第三次握手：客户端收到服务器的SYN+ACK报文后，会发送一个确认报文给服务器，报文包含ACK=1，seq=x+1（客户端序列号加1），ack=y+1（确认服务器的序列号加1）。此时，TCP连接建立成功，客户端和服务器都进入ESTABLISHED状态，可以开始传输数据。
  • 四次挥手（Four-Way Handshake）
    • 第一次挥手：当客户端完成数据传输后，它会发送一个FIN（Finish）标志置位的报文给服务器，告诉服务器自己已经没有数据要发送了，但是还可以接收数据。客户端进入FIN_WAIT_1状态。
    • 第二次挥手：服务器收到客户端的FIN报文后，会发送一个ACK确认报文给客户端，告诉客户端自己已经知道客户端没有数据发送了，但服务器可能还有数据要发送给客户端，所以连接没有立即关闭。此时服务器进入CLOSE_WAIT状态，客户端进入FIN_WAIT_2状态。
    • 第三次挥手：当服务器也完成数据传输后，会发送一个带有FIN标志的报文给客户端，告诉客户端自己也没有数据要发送了，请求关闭连接。服务器进入LAST_ACK状态。
    • 第四次挥手：客户端收到服务器的FIN报文后，会发送一个ACK报文给服务器，确认收到了服务器的关闭请求。此时客户端进入TIME_WAIT状态，等待足够的时间确保最后一个ACK报文能够到达服务器（这个时间通常是2MSL，即两倍的最大段生存期），之后客户端关闭连接。服务器收到客户端的ACK后，关闭连接。
  • 通过这些握手和挥手的过程，TCP确保了连接的可靠建立和有序、干净地关闭。

  • 强制缓存和协商缓存 ​

强制缓存（Freshness Cache）和协商缓存（Validation Cache）是HTTP协议中两种主要的缓存机制，它们帮助减少网络延迟，提高页面加载速度。

强制缓存（Freshness Cache） 强制缓存不需要与服务器通信，浏览器直接使用缓存副本，除非资源过期。主要通过以下两个响应头来控制：

Expires：这是一个日期/时间头，表示资源在何时之前被认为是新鲜的，过了这个时间则被认为过期。但这个头受到时区和服务器时间的影响，可能会有误差。 Cache-Control：更现代的控制头，可以设置多个指令，如max-age（资源在多少秒内被认为是新鲜的），s-maxage（针对代理服务器的max-age），以及public或private（是否可被共享缓存）等。 当浏览器发现资源在缓存中且未过期时，会直接使用缓存副本，跳过服务器验证。

协商缓存（Validation Cache） 协商缓存需要与服务器进行通信，验证缓存资源是否是最新的。浏览器会发送请求，携带以下请求头：

If-Modified-Since：包含资源上一次请求时的响应头中的Last-Modified日期，询问服务器资源是否在此日期后有改动。 If-None-Match：包含资源上一次请求时的响应头中的ETag（实体标签），这是一个唯一标识资源的值，用于判断资源是否改变。 服务器接收到请求后，对比这些信息，如果资源未变，会返回304 Not Modified状态码，告知浏览器继续使用缓存副本。如果资源已更新，服务器会返回新的资源和相应的响应头。

工作流程 浏览器首次请求资源，服务器返回资源和相关缓存头（Expires、Cache-Control、Last-Modified、ETag等）。 浏览器缓存资源和头信息。 下次请求同资源时，浏览器检查强制缓存。如果资源未过期，直接使用缓存；否则，进入协商缓存。 浏览器发送带有If-Modified-Since或If-None-Match的请求。 服务器根据头信息判断资源是否更新，返回相应状态码和资源（如有更新）。 通过这两种缓存机制，浏览器可以有效利用本地缓存，减少不必要的网络请求，提高用户体验。

谈谈 ajax ​

可以发送异步请求，进行局部页面更新

流程：

  1. new XMLHttpRequest() 来创建 xhr 对象
  2. 通过 xhr 调用 open 方法，设置请求方式和请求地址（请求地址可以添加查询字符串参数）
  3. 通过 xhr.send 方法发送请求（可以携带请求体参数）
  4. 通过 xhr.onreadystatechange 或 onload 事件监听响应回来的结果，往往需要判断 xhr 的响应状态码是否 2 开头，来判断请求成功还是失败

跨域 ​

  1. 什么是跨域？ 违背同源策略产生跨域。

  2. 什么是同源策略？ 同源指的是：协议、域名、端口号三者必须完全一致。

  3. 什么样的请求存在跨域问题？

只有浏览器端的 ajax 请求存在跨域。

服务器与服务器之间没有跨域，浏览器的 script、img、form 等标签都没有跨域。

  1. 解决跨域的方案：
  • jsonp: 利用 script 标签可以跨域特性进行跨域。

可以说说具体做法

  • cors: 服务器设置特定响应头即可实现。access-control-allow-Origin

  • NodeJS 代理服务器: 这是我开发中主要使用解决跨域的方案

比如 vue2 项目，可以在 vue.config.js 中进行配置，配置 devServer 的 proxy，设置代理请求前缀和目标服务器地址等来完成

  • Nginx 代理服务器：项目上线时公司采用 nginx 反向代理解决跨域
